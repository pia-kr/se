# 1. 서론

## 1.1 문서의 목적 및 범위

본 문서는 **Mini Drive 프로젝트**의 체계적인 수행과 효율적인 관리를 위해 작성되었다.

### 목적

- 프로젝트 목표 및 범위 명확화
- 일정 및 역할 분담 정의
- 개발 절차 및 품질 관리 기준 수립
- 팀원 간 원활한 의사소통 지원
- 성공적인 프로젝트 완료 지원

### 범위

Mini Drive는 조직 내부에서 사용하는 **웹 기반 클라우드 파일 공유 시스템**으로 다음 기능을 포함한다.

- 파일 업로드 / 다운로드
- 폴더 구조 관리
- 사용자 간 공유 및 권한 설정
- 외부 링크 공유
- 파일 검색
- 버전 관리
- 관리자 기능

---

## 1.2 프로젝트 개요

기존에는 이메일, 메신저, 개인 클라우드 등 여러 경로로 파일을 관리하여 다음과 같은 문제가 발생하였다.

- 최신 파일 버전 확인 어려움
- 자료 검색 시간 증가
- 공유 권한 통제 어려움
- 외부 공유 시 보안 문제 발생

Mini Drive는 이를 해결하기 위해 중앙 집중형 파일 관리 시스템을 구축한다.

# 2. 개발 계획

## 2.1 개발 절차 모형

본 프로젝트는 **Agile 방법론 중 Scrum 모델**을 적용한다.

### 적용 이유

- 요구사항 변경 가능성 존재
- 기능 단위 반복 개발 필요
- 빠른 피드백 반영 가능
- 팀 협업 효율성 향상

### 운영 방식

- Sprint 단위 개발 (1~2주)
- Daily Scrum 진행
- Sprint Review 수행
- Sprint Retrospective 수행

# 3. 인력 구성

## 3.1 팀 구조

본 프로젝트는 프로젝트 관리자를 중심으로 운영되며,  
기능별 전문 인력을 배치한 **협업 중심 조직 구조**를 따른다.

---

## 3.2 역할 및 책임

| 역할 | 인원 | 주요 책임 |
|------|------|----------|
| 프로젝트 관리자 (PM) | 1명 | 일정 관리, 업무 조율, 전체 진행 총괄 |
| 프론트엔드 개발자 | 2명 | UI/UX 설계 및 화면 기능 구현 |
| 백엔드 개발자 | 2명 | 서버 로직, API 개발, 인증 기능 구현 |
| DB 관리자 (DBA) | 1명 | 데이터베이스 설계, 성능 최적화, 백업 관리 |
| 품질 관리 담당 (QA) | 1명 | 테스트 계획 수립, 기능 검증, 오류 관리 |
| 문서 담당 | 1명 | 산출물 문서 작성, 발표자료 제작 |

---

## 3.3 공통 책임

- 주간 회의 및 Sprint 진행 상황 공유
- GitHub 기반 형상관리 및 협업 수행
- 기능 테스트 및 오류 수정 지원
- 프로젝트 산출물 공동 검토
- 최종 발표 및 시연 준비
